Build da Imagem Docker

O processo de build é o passo inicial do fluxo de deploy. O objetivo é gerar uma imagem enxuta, segura e reproduzível, pronta para ser executada no Kubernetes.

Estrutura do Dockerfile​

O padrão recomendado para produção é o multi-stage build. Ele separa o ambiente de compilação/build do ambiente de execução final, resultando em imagens menores e sem dependências desnecessárias.

Exemplo: Aplicação Node.js​

# ─── Stage 1: Build ───────────────────────────────────────────────
FROM node:20-alpine AS builder

WORKDIR /app

# Copia apenas os arquivos de dependências primeiro (melhor uso de cache)
COPY package.json package-lock.json ./
RUN npm ci --only=production

# Copia o restante do código e executa o build
COPY . .
RUN npm run build

# ─── Stage 2: Runtime ─────────────────────────────────────────────
FROM node:20-alpine AS runtime

# Não executar como root em produção
RUN addgroup -S appgroup && adduser -S appuser -G appgroup

WORKDIR /app

# Copia apenas os artefatos necessários do stage de build
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
COPY package.json ./

USER appuser

EXPOSE 3000
CMD ["node", "dist/main.js"]

Exemplo: Aplicação Go​

# ─── Stage 1: Build ───────────────────────────────────────────────
FROM golang:1.22-alpine AS builder

WORKDIR /app

COPY go.mod go.sum ./
RUN go mod download

COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -ldflags="-s -w" -o server ./cmd/server

# ─── Stage 2: Runtime ─────────────────────────────────────────────
FROM scratch

COPY --from=builder /app/server /server
COPY --from=builder /etc/ssl/certs/ca-certificates.crt /etc/ssl/certs/

EXPOSE 8080
ENTRYPOINT ["/server"]

O uso de FROM scratch ou alpine no stage final reduz drasticamente o tamanho da imagem e a superfície de ataque em produção.

Arquivo .dockerignore​

Sempre utilize um .dockerignore para evitar copiar arquivos desnecessários para o contexto de build, acelerando o processo e evitando vazamento de arquivos sensíveis.

# Dependências locais
node_modules/
vendor/

# Arquivos de ambiente e segredos
.env
.env.*
*.pem
*.key

# Artefatos de build local
dist/
build/
out/

# Controle de versão
.git/
.gitignore

# Arquivos de editor e OS
.DS_Store
.vscode/
*.log

# Docker
Dockerfile*
docker-compose*

Executando o Build​

Build básico​

docker build -t minha-app:1.0.0 .

Build com argumentos (build args)​

Use ARG no Dockerfile para parametrizar valores em tempo de build:

docker build \
  --build-arg NODE_ENV=production \
  --build-arg APP_VERSION=1.0.0 \
  -t minha-app:1.0.0 \
  .
ARG NODE_ENV=development
ARG APP_VERSION=local

ENV NODE_ENV=$NODE_ENV
ENV APP_VERSION=$APP_VERSION

Build com plataforma específica (multi-arch)​

Para clusters que rodam em ARM (ex: AWS Graviton, Apple Silicon):

docker buildx build \
  --platform linux/amd64,linux/arm64 \
  -t minha-app:1.0.0 \
  --push \
  .

Em pipelines CI/CD, sempre especifique a plataforma de destino (--platform linux/amd64) para evitar incompatibilidade com os nós do cluster Kubernetes.

Convenção de Tags​

Utilize uma convenção de tags consistente para facilitar rastreabilidade e rollback:

Padrão de TagQuando usar
1.0.0Versão semântica para releases estáveis
1.0.0-rc.1Release candidate para testes
main-abc1234Branch + hash do commit (útil em CI/CD)
latestÚltima versão estável (evitar em produção)
# Boas práticas de tagging
IMAGE_TAG=$(git rev-parse --short HEAD)
APP_VERSION="1.0.0"

docker build -t minha-app:${APP_VERSION} .
docker tag minha-app:${APP_VERSION} minha-app:main-${IMAGE_TAG}

Evite usar apenas latest em produção. Tags imutáveis como a versão semântica ou o hash do commit garantem reprodutibilidade e facilitam rollbacks.

Verificando a Imagem​

Após o build, inspecione a imagem antes de enviá-la:

# Ver detalhes e tamanho
docker image inspect minha-app:1.0.0

# Listar camadas e tamanhos
docker history minha-app:1.0.0

# Testar localmente antes do push
docker run --rm -p 3000:3000 minha-app:1.0.0
