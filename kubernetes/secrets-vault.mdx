# HashiCorp Vault no Kubernetes

O **HashiCorp Vault** é um cofre de segredos centralizado que oferece armazenamento criptografado, controle de acesso por política e auditoria completa. A integração com Kubernetes permite que Pods recuperem segredos diretamente do Vault sem expor credenciais em Secrets nativos do Kubernetes.

## Arquitetura de Integração

```
GitLab CI/CD ──► JWT Auth ──► Vault ──► segredos na pipeline
                                │
Kubernetes Pod ──► ServiceAccount Token ──► Vault ──► segredos injetados via:
                                              ├── Vault Agent Injector (sidecar)
                                              ├── Vault CSI Provider (volume)
                                              └── External Secrets Operator
```

## Instalação do Vault no Kubernetes

```bash
helm repo add hashicorp https://helm.releases.hashicorp.com
helm repo update

helm install vault hashicorp/vault \
  --namespace vault \
  --create-namespace \
  --set server.ha.enabled=true \
  --set server.ha.replicas=3 \
  --set server.ingress.enabled=true \
  --set server.ingress.hosts[0].host=vault.empresa.com.br \
  --set injector.enabled=true        # Habilita o Vault Agent Injector
```

### Inicializar e Desselar

```bash
# Inicializar (gera as unseal keys e root token — guarde com segurança!)
kubectl exec -n vault vault-0 -- vault operator init \
  -key-shares=5 \
  -key-threshold=3 \
  -format=json > vault-init.json

# Desselar cada pod (use 3 das 5 chaves geradas)
for i in 0 1 2; do
  kubectl exec -n vault vault-$i -- vault operator unseal $(jq -r '.unseal_keys_b64[0]' vault-init.json)
  kubectl exec -n vault vault-$i -- vault operator unseal $(jq -r '.unseal_keys_b64[1]' vault-init.json)
  kubectl exec -n vault vault-$i -- vault operator unseal $(jq -r '.unseal_keys_b64[2]' vault-init.json)
done

# Verificar status
kubectl exec -n vault vault-0 -- vault status
```

<Warning>
  As unseal keys e o root token gerados no `vault operator init` devem ser armazenados com segurança (ex: cofre físico, HSM, Vault Enterprise Auto Unseal). Nunca commite esses valores em repositórios.
</Warning>

## Configurar Auth Method Kubernetes

Permite que Pods se autentiquem no Vault usando o ServiceAccount Token do Kubernetes:

```bash
# Exportar variáveis de conexão
export VAULT_ADDR="https://vault.empresa.com.br"
export VAULT_TOKEN=$(jq -r '.root_token' vault-init.json)

# Habilitar o método de autenticação Kubernetes
vault auth enable kubernetes

# Configurar com as credenciais do cluster
vault write auth/kubernetes/config \
  kubernetes_host="https://$(kubectl get svc kubernetes -o jsonpath='{.spec.clusterIP}'):443" \
  kubernetes_ca_cert=@<(kubectl config view --raw --minify --flatten -o jsonpath='{.clusters[].cluster.certificate-authority-data}' | base64 -d) \
  token_reviewer_jwt=@<(kubectl create token vault-auth -n vault)
```

## Criar Políticas e Segredos

```bash
# Habilitar o engine de segredos KV v2
vault secrets enable -path=producao kv-v2

# Criar segredos
vault kv put producao/minha-app \
  db_password="senha-super-secreta" \
  db_username="app_user" \
  api_key="chave-api-externa-123" \
  jwt_secret="meu-jwt-secret-producao"

# Criar política de acesso (somente leitura para a aplicação)
vault policy write minha-app-policy - <<EOF
path "producao/data/minha-app" {
  capabilities = ["read"]
}

path "producao/data/minha-app/*" {
  capabilities = ["read"]
}
EOF

# Criar role Kubernetes vinculando ServiceAccount à política
vault write auth/kubernetes/role/minha-app-role \
  bound_service_account_names=minha-app-sa \
  bound_service_account_namespaces=producao \
  policies=minha-app-policy \
  ttl=1h
```

```bash
# Criar o ServiceAccount no namespace
kubectl create serviceaccount minha-app-sa --namespace=producao
```

## Método 1: Vault Agent Injector (Sidecar via Anotações)

O **Vault Agent Injector** monitora Pods com anotações específicas e injeta automaticamente um container sidecar que busca os segredos e os disponibiliza como arquivos:

```yaml
# deployment-vault-injector.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: minha-app
  namespace: producao
spec:
  replicas: 2
  selector:
    matchLabels:
      app: minha-app
  template:
    metadata:
      labels:
        app: minha-app
      annotations:
        # ── Anotações do Vault Agent Injector ─────────────────────
        vault.hashicorp.com/agent-inject: "true"
        vault.hashicorp.com/role: "minha-app-role"
        vault.hashicorp.com/agent-pre-populate-only: "false"

        # Injeta o segredo como arquivo em /vault/secrets/app-secrets
        vault.hashicorp.com/agent-inject-secret-app-secrets: "producao/data/minha-app"

        # Template para formatar o arquivo injetado
        vault.hashicorp.com/agent-inject-template-app-secrets: |
          {{- with secret "producao/data/minha-app" -}}
          DB_PASSWORD={{ .Data.data.db_password }}
          DB_USERNAME={{ .Data.data.db_username }}
          API_KEY={{ .Data.data.api_key }}
          JWT_SECRET={{ .Data.data.jwt_secret }}
          {{- end }}

        # Renovação automática quando o token expira
        vault.hashicorp.com/agent-inject-token: "false"
        vault.hashicorp.com/agent-revoke-on-shutdown: "true"
        vault.hashicorp.com/tls-skip-verify: "false"
    spec:
      serviceAccountName: minha-app-sa   # ServiceAccount vinculado ao role Vault
      containers:
        - name: minha-app
          image: registry.gitlab.com/empresa/minha-app:1.0.0
          command: ["/bin/sh", "-c"]
          args:
            # Carrega o arquivo de segredos injetado pelo Vault Agent
            - |
              source /vault/secrets/app-secrets
              exec node dist/main.js
          ports:
            - containerPort: 3000
          resources:
            requests:
              cpu: "250m"
              memory: "256Mi"
            limits:
              cpu: "500m"
              memory: "512Mi"
```

O Vault Agent Injector adiciona automaticamente dois containers init/sidecar ao Pod:

```
Pod: minha-app
  ├── vault-agent-init   (init container — busca segredos na inicialização)
  ├── vault-agent        (sidecar — renova segredos periodicamente)
  └── minha-app          (seu container — lê de /vault/secrets/)
```

## Método 2: Vault CSI Provider (Volume Secreto)

Monta os segredos como um volume no sistema de arquivos do Pod, usando a interface CSI:

```bash
# Instalar o Secrets Store CSI Driver
helm repo add secrets-store-csi-driver https://kubernetes-sigs.github.io/secrets-store-csi-driver/charts
helm install csi-secrets-store secrets-store-csi-driver/secrets-store-csi-driver \
  --namespace kube-system \
  --set syncSecret.enabled=true        # Sincroniza para Kubernetes Secret também

# Instalar o provider do Vault
helm install vault-csi-provider hashicorp/vault \
  --namespace vault \
  --set "injector.enabled=false" \
  --set "csi.enabled=true"
```

```yaml
# secretproviderclass-vault.yaml
apiVersion: secrets-store.csi.x-k8s.io/v1
kind: SecretProviderClass
metadata:
  name: vault-minha-app
  namespace: producao
spec:
  provider: vault
  parameters:
    vaultAddress: "https://vault.empresa.com.br"
    roleName: "minha-app-role"
    objects: |
      - objectName: "db_password"
        secretPath: "producao/data/minha-app"
        secretKey: "db_password"
      - objectName: "db_username"
        secretPath: "producao/data/minha-app"
        secretKey: "db_username"
      - objectName: "api_key"
        secretPath: "producao/data/minha-app"
        secretKey: "api_key"

  # Opcional: sincroniza como Kubernetes Secret
  secretObjects:
    - secretName: minha-app-vault-sync
      type: Opaque
      data:
        - objectName: "db_password"
          key: DB_PASSWORD
        - objectName: "api_key"
          key: API_KEY
```

```yaml
# deployment-vault-csi.yaml
spec:
  template:
    spec:
      serviceAccountName: minha-app-sa
      containers:
        - name: minha-app
          image: registry.gitlab.com/empresa/minha-app:1.0.0
          env:
            # Lê do Kubernetes Secret sincronizado pelo CSI
            - name: DB_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: minha-app-vault-sync
                  key: DB_PASSWORD
          volumeMounts:
            - name: vault-secrets
              mountPath: /mnt/secrets
              readOnly: true
      volumes:
        - name: vault-secrets
          csi:
            driver: secrets-store.csi.k8s.io
            readOnly: true
            volumeAttributes:
              secretProviderClass: vault-minha-app
```

## Método 3: External Secrets Operator (ESO)

Abordagem mais declarativa e desacoplada — cria Kubernetes Secrets nativos sincronizados periodicamente com o Vault:

```bash
helm repo add external-secrets https://charts.external-secrets.io
helm install external-secrets external-secrets/external-secrets \
  --namespace external-secrets \
  --create-namespace
```

```yaml
# vault-secretstore.yaml
apiVersion: external-secrets.io/v1beta1
kind: SecretStore
metadata:
  name: vault-backend
  namespace: producao
spec:
  provider:
    vault:
      server: "https://vault.empresa.com.br"
      path: "producao"
      version: "v2"
      auth:
        kubernetes:
          mountPath: "kubernetes"
          role: "minha-app-role"
          serviceAccountRef:
            name: minha-app-sa
```

```yaml
# externalsecret-minha-app.yaml
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: minha-app-secrets
  namespace: producao
spec:
  refreshInterval: 5m          # Sincroniza a cada 5 minutos
  secretStoreRef:
    name: vault-backend
    kind: SecretStore
  target:
    name: minha-app-vault-secret   # Nome do Kubernetes Secret gerado
    creationPolicy: Owner
    deletionPolicy: Retain
  data:
    - secretKey: DB_PASSWORD
      remoteRef:
        key: minha-app
        property: db_password
    - secretKey: DB_USERNAME
      remoteRef:
        key: minha-app
        property: db_username
    - secretKey: API_KEY
      remoteRef:
        key: minha-app
        property: api_key
    - secretKey: JWT_SECRET
      remoteRef:
        key: minha-app
        property: jwt_secret
```

O ESO cria um Kubernetes Secret padrão `minha-app-vault-secret` que pode ser usado normalmente em qualquer Deployment com `envFrom` ou `secretKeyRef`.

## Integração com GitLab CI/CD

### Configurar JWT Auth no Vault para GitLab

O GitLab emite um **JWT token** por job. Configure o Vault para aceitar esse token como autenticação:

```bash
# Habilitar JWT auth
vault auth enable jwt

# Configurar com o JWKS público do GitLab
vault write auth/jwt/config \
  jwks_url="https://gitlab.empresa.com.br/-/jwks" \
  bound_issuer="https://gitlab.empresa.com.br"

# Criar política para pipelines (somente leitura)
vault policy write gitlab-ci-policy - <<EOF
path "producao/data/minha-app" {
  capabilities = ["read"]
}

path "producao/data/shared/*" {
  capabilities = ["read"]
}
EOF

# Criar role JWT vinculando ao projeto GitLab
vault write auth/jwt/role/gitlab-minha-app \
  role_type="jwt" \
  policies="gitlab-ci-policy" \
  token_explicit_max_ttl="60m" \
  user_claim="user_email" \
  bound_claims_type="glob" \
  bound_claims="{
    \"project_path\": \"empresa/minha-app\",
    \"ref_protected\": \"true\",
    \"ref_type\": \"branch\",
    \"ref\": \"main\"
  }"
```

<Note>
  Os `bound_claims` restringem qual projeto, branch e tipo de ref podem obter o token. Use `"ref": "*"` para permitir qualquer branch, mas prefira restringir a branches protegidas em produção.
</Note>

### `.gitlab-ci.yml` — Buscar segredos do Vault

```yaml
# .gitlab-ci.yml
stages:
  - test
  - build
  - deploy

variables:
  VAULT_ADDR: "https://vault.empresa.com.br"
  VAULT_ROLE: "gitlab-minha-app"

# ── Job reutilizável para autenticação no Vault ────────────────────
.vault-auth: &vault-auth
  before_script:
    - |
      # Autenticar no Vault com o JWT do job
      VAULT_TOKEN=$(curl -sf \
        --request POST \
        --data "{\"jwt\": \"${CI_JOB_JWT_V2}\", \"role\": \"${VAULT_ROLE}\"}" \
        ${VAULT_ADDR}/v1/auth/jwt/login | jq -r '.auth.client_token')

      if [ -z "$VAULT_TOKEN" ] || [ "$VAULT_TOKEN" = "null" ]; then
        echo "Falha ao autenticar no Vault"
        exit 1
      fi

      export VAULT_TOKEN

      # Buscar segredos e exportar como variáveis de ambiente
      SECRETS=$(curl -sf \
        -H "X-Vault-Token: ${VAULT_TOKEN}" \
        ${VAULT_ADDR}/v1/producao/data/minha-app | jq -r '.data.data')

      export DB_PASSWORD=$(echo $SECRETS | jq -r '.db_password')
      export DB_USERNAME=$(echo $SECRETS | jq -r '.db_username')
      export API_KEY=$(echo $SECRETS     | jq -r '.api_key')
      export JWT_SECRET=$(echo $SECRETS  | jq -r '.jwt_secret')

# ── Ou usando o CLI oficial do Vault ──────────────────────────────
.vault-auth-cli: &vault-auth-cli
  image: hashicorp/vault:1.16
  before_script:
    - |
      vault login \
        -method=jwt \
        -path=jwt \
        role=${VAULT_ROLE} \
        jwt=${CI_JOB_JWT_V2}

      # Exportar como variáveis de ambiente via vault kv get
      eval $(vault kv get \
        -format=json producao/minha-app \
        | jq -r '.data.data | to_entries[] | "export \(.key | ascii_upcase)=\(.value)"')

# ── Jobs da pipeline ──────────────────────────────────────────────
unit-tests:
  stage: test
  image: node:20-alpine
  <<: *vault-auth
  script:
    - npm ci
    - npm test
  variables:
    NODE_ENV: test

build-image:
  stage: build
  image: docker:24
  services:
    - docker:24-dind
  variables:
    DOCKER_TLS_CERTDIR: "/certs"
  before_script:
    # Autenticação no Vault para buscar credenciais do registry privado
    - |
      VAULT_TOKEN=$(curl -sf \
        --request POST \
        --data "{\"jwt\": \"${CI_JOB_JWT_V2}\", \"role\": \"${VAULT_ROLE}\"}" \
        ${VAULT_ADDR}/v1/auth/jwt/login | jq -r '.auth.client_token')
      export VAULT_TOKEN

      REGISTRY_PASS=$(curl -sf \
        -H "X-Vault-Token: ${VAULT_TOKEN}" \
        ${VAULT_ADDR}/v1/producao/data/registry \
        | jq -r '.data.data.password')

      docker login registry.empresa.local:5000 \
        --username deploy \
        --password "${REGISTRY_PASS}"
  script:
    - docker build -t $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA .
    - docker push $CI_REGISTRY_IMAGE:$CI_COMMIT_SHORT_SHA
  rules:
    - if: $CI_COMMIT_BRANCH == "main"

deploy-producao:
  stage: deploy
  image: bitnami/kubectl:latest
  <<: *vault-auth
  script:
    - |
      # Atualizar o Secret do Kubernetes com os valores frescos do Vault
      kubectl create secret generic minha-app-secrets \
        --from-literal=DB_PASSWORD="${DB_PASSWORD}" \
        --from-literal=DB_USERNAME="${DB_USERNAME}" \
        --from-literal=API_KEY="${API_KEY}" \
        --namespace=producao \
        --dry-run=client -o yaml | kubectl apply -f -

      # Aplicar manifests e aguardar rollout
      kubectl apply -f k8s/ --namespace=producao
      kubectl rollout status deployment/minha-app --namespace=producao --timeout=5m
  environment:
    name: producao
  rules:
    - if: $CI_COMMIT_BRANCH == "main"
      when: manual
```

### Usando o provider nativo do GitLab para Vault (GitLab 14.9+)

Para GitLab com integração nativa ao Vault, declare os segredos diretamente no job:

```yaml
# .gitlab-ci.yml (com provider nativo do GitLab)
deploy-producao:
  stage: deploy
  image: bitnami/kubectl:latest
  secrets:
    DB_PASSWORD:
      vault: producao/minha-app/db_password@producao   # path/secret/field@engine
      file: false
    API_KEY:
      vault: producao/minha-app/api_key@producao
      file: false
  script:
    - echo "DB_PASSWORD está disponível como variável de ambiente"
    - kubectl apply -f k8s/ --namespace=producao
  rules:
    - if: $CI_COMMIT_BRANCH == "main"
```

Configure a URL do Vault no GitLab em **Settings → CI/CD → Variables** ou **Admin → Settings → CI/CD → Vault server**.

## Rotação de Segredos

O Vault suporta **Dynamic Secrets** — gera credenciais únicas por job com TTL curto, que expiram automaticamente:

```bash
# Habilitar engine de banco de dados
vault secrets enable database

# Configurar conexão com PostgreSQL
vault write database/config/postgres \
  plugin_name=postgresql-database-plugin \
  allowed_roles="app-role" \
  connection_url="postgresql://{{username}}:{{password}}@postgres:5432/producao" \
  username="vault_admin" \
  password="senha-do-vault-admin"

# Criar role que gera credenciais temporárias
vault write database/roles/app-role \
  db_name=postgres \
  creation_statements="CREATE ROLE \"{{name}}\" WITH LOGIN PASSWORD '{{password}}' VALID UNTIL '{{expiration}}'; GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO \"{{name}}\";" \
  default_ttl="1h" \
  max_ttl="24h"
```

```bash
# Cada job/Pod gera suas próprias credenciais únicas
vault read database/creds/app-role
# Key                Value
# username           v-gitl-minha-app-AbCdEf123
# password           A1B2C3D4E5F6...
# lease_duration     1h
```

## Comparativo dos Métodos de Integração

| Método                  | Como funciona                        | Kubernetes Secret | Reinicia Pod para atualizar | Melhor para            |
|-------------------------|--------------------------------------|:-----------------:|:---------------------------:|------------------------|
| Vault Agent Injector    | Sidecar injeta arquivos em `/vault/secrets/` | Não        | Não (renovação automática)  | Produção geral         |
| Vault CSI Provider      | Volume CSI montado no Pod            | Opcional           | Não                         | Acesso via arquivo      |
| External Secrets Operator | Cria/atualiza Kubernetes Secret    | Sim               | Sim (se mudar o Secret)      | Integração com GitOps  |
