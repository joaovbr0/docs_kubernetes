ConfigMap e Secret

Separar configuração do código é um dos princípios fundamentais de aplicações cloud-native (12-Factor App). No Kubernetes, isso é feito por dois recursos:

file-code
ConfigMap

Armazena configurações não sensíveis: URLs, nomes de banco, flags de feature, parâmetros de aplicação.

lock
Secret

Armazena dados sensíveis codificados em base64: senhas, tokens, certificados, chaves de API.

Secrets no Kubernetes são codificados em base64, não criptografados. Em produção, use soluções complementares como Sealed Secrets, External Secrets Operator com AWS Secrets Manager/HashiCorp Vault, ou habilite Encryption at Rest no etcd.

ConfigMap​

Criar via manifesto YAML​

# configmap-app.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
  namespace: producao
data:
  # Chave/valor simples
  NODE_ENV: "production"
  PORT: "3000"
  LOG_LEVEL: "info"
  APP_NAME: "minha-app"
  DB_HOST: "postgres-service.producao.svc.cluster.local"
  DB_PORT: "5432"
  DB_NAME: "producao"
  REDIS_HOST: "redis-service.producao.svc.cluster.local"
  REDIS_PORT: "6379"

  # Valor multilinha (arquivo de configuração)
  app.properties: |
    max.connections=100
    timeout=30000
    retry.attempts=3
    cache.ttl=3600

  nginx.conf: |
    server {
      listen 80;
      location / {
        proxy_pass http://localhost:3000;
      }
    }
kubectl apply -f configmap-app.yaml

# Criar via CLI (simples)
kubectl create configmap app-config \
  --from-literal=NODE_ENV=production \
  --from-literal=PORT=3000 \
  --namespace=producao

# Criar a partir de um arquivo .env
kubectl create configmap app-config \
  --from-env-file=.env.production \
  --namespace=producao

# Criar a partir de um arquivo de configuração
kubectl create configmap nginx-config \
  --from-file=nginx.conf \
  --namespace=producao

Injetar ConfigMap como variáveis de ambiente​

spec:
  containers:
    - name: minha-app
      image: registry.gitlab.com/empresa/minha-app:1.0.0

      # Opção 1: Carregar TODAS as chaves do ConfigMap como variáveis
      envFrom:
        - configMapRef:
            name: app-config

      # Opção 2: Selecionar chaves específicas
      env:
        - name: NODE_ENV
          valueFrom:
            configMapKeyRef:
              name: app-config
              key: NODE_ENV
        - name: DB_HOST
          valueFrom:
            configMapKeyRef:
              name: app-config
              key: DB_HOST

Injetar ConfigMap como arquivo (volume)​

spec:
  containers:
    - name: nginx
      image: nginx:1.25-alpine
      volumeMounts:
        - name: nginx-config-volume
          mountPath: /etc/nginx/conf.d
          readOnly: true

  volumes:
    - name: nginx-config-volume
      configMap:
        name: nginx-config
        items:
          - key: nginx.conf
            path: default.conf      # Nome do arquivo dentro do container

Quando um ConfigMap é atualizado, os arquivos montados via volume são atualizados automaticamente (com alguns segundos de delay). Variáveis de ambiente não são atualizadas — o Pod precisa ser reiniciado.

Secret​

Criar via manifesto YAML​

Os valores devem estar em base64:

# Gerar valores em base64
echo -n "minha-senha-segura" | base64
# bWluaGEtc2VuaGEtc2VndXJh

echo -n "usuario" | base64
# dXN1YXJpbw==
# secret-db.yaml
apiVersion: v1
kind: Secret
metadata:
  name: db-secret
  namespace: producao
type: Opaque
data:
  username: dXN1YXJpbw==               # usuario
  password: bWluaGEtc2VuaGEtc2VndXJh  # minha-senha-segura
  url: cG9zdGdyZXM6Ly91c3Vhcmlv...    # postgres://usuario:senha@host/db

Criar via CLI (forma mais segura — não fica no histórico)​

# Criar secret com valores diretamente
kubectl create secret generic db-secret \
  --from-literal=username=usuario \
  --from-literal=password=minha-senha-segura \
  --namespace=producao

# Criar a partir de arquivos (útil para certificados TLS)
kubectl create secret generic tls-certs \
  --from-file=tls.crt=./certs/server.crt \
  --from-file=tls.key=./certs/server.key \
  --namespace=producao

# Criar Secret TLS com tipo específico
kubectl create secret tls minha-app-tls \
  --cert=./certs/server.crt \
  --key=./certs/server.key \
  --namespace=producao

Tipos de Secret​

TipoUso
OpaqueDados arbitrários (padrão)
kubernetes.io/tlsCertificados TLS
kubernetes.io/dockerconfigjsonCredenciais de registry Docker
kubernetes.io/service-account-tokenToken de ServiceAccount
kubernetes.io/ssh-authChaves SSH

Injetar Secret como variáveis de ambiente​

spec:
  containers:
    - name: minha-app
      image: registry.gitlab.com/empresa/minha-app:1.0.0

      # Opção 1: Todas as chaves do Secret
      envFrom:
        - secretRef:
            name: db-secret

      # Opção 2: Chaves específicas
      env:
        - name: DB_PASSWORD
          valueFrom:
            secretKeyRef:
              name: db-secret
              key: password
        - name: DB_USERNAME
          valueFrom:
            secretKeyRef:
              name: db-secret
              key: username

Injetar Secret como arquivo (volume)​

Ideal para certificados e chaves que precisam ser arquivos:

spec:
  containers:
    - name: minha-app
      volumeMounts:
        - name: certs
          mountPath: /app/certs
          readOnly: true

  volumes:
    - name: certs
      secret:
        secretName: tls-certs
        defaultMode: 0400       # Permissão restrita para o arquivo

Combinando ConfigMap e Secret no Deployment​

apiVersion: apps/v1
kind: Deployment
metadata:
  name: minha-app
  namespace: producao
spec:
  replicas: 3
  selector:
    matchLabels:
      app: minha-app
  template:
    metadata:
      labels:
        app: minha-app
    spec:
      containers:
        - name: minha-app
          image: registry.gitlab.com/empresa/minha-app:1.0.0
          envFrom:
            - configMapRef:
                name: app-config       # Variáveis não sensíveis
            - secretRef:
                name: app-secrets      # Variáveis sensíveis
          volumeMounts:
            - name: app-config-files
              mountPath: /app/config
              readOnly: true
            - name: tls-certs
              mountPath: /app/certs
              readOnly: true

      volumes:
        - name: app-config-files
          configMap:
            name: app-config
        - name: tls-certs
          secret:
            secretName: minha-app-tls
            defaultMode: 0400

Boas Práticas de Segurança​

Nunca commitar Secrets no Git›

Nunca inclua Secrets (nem em base64) em repositórios Git. Use .gitignore para arquivos com segredos e prefira criar Secrets via CLI ou ferramentas dedicadas.

Use External Secrets Operator›

Integre com cofres de segredos externos (AWS Secrets Manager, Azure Key Vault, HashiCorp Vault) para que os Secrets sejam gerenciados fora do cluster:

apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: db-secret
  namespace: producao
spec:
  refreshInterval: 1h
  secretStoreRef:
    name: aws-secret-store
    kind: SecretStore
  target:
    name: db-secret
  data:
    - secretKey: password
      remoteRef:
        key: prod/minha-app/db
        property: password
Habilite Encryption at Rest no etcd›

Por padrão, Secrets são armazenados em texto puro no etcd. Configure EncryptionConfiguration no API Server para criptografar em repouso.

Restrinja acesso via RBAC›

Use Role e RoleBinding para limitar quais ServiceAccounts podem ler Secrets de produção.

apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: secret-reader
  namespace: producao
rules:
  - apiGroups: [""]
    resources: ["secrets"]
    resourceNames: ["db-secret"]   # Apenas este Secret específico
    verbs: ["get"]

Comandos de Referência​

# Listar ConfigMaps
kubectl get configmaps -n producao

# Ver conteúdo de um ConfigMap
kubectl get configmap app-config -n producao -o yaml

# Editar um ConfigMap
kubectl edit configmap app-config -n producao

# Listar Secrets (os valores não são exibidos)
kubectl get secrets -n producao

# Ver um Secret decodificado
kubectl get secret db-secret -n producao -o jsonpath='{.data.password}' | base64 -d

# Deletar e recriar um Secret
kubectl delete secret db-secret -n producao
kubectl create secret generic db-secret \
  --from-literal=password=nova-senha \
  --namespace=producao
